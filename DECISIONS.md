# DECISIONS.md — Telegram Bots

Architectural and product decisions with reasoning.

| Timestamp (MYT) | Decision | Choice | Reasoning |
|---|---|---|---|
| 2026-02-24 12:10 | Bot framework: grammY or node-telegram-bot-api? | grammY v1.40.0 | grammY is more actively maintained, has better modern JS support, built-in webhook adapter for Express, first-class inline keyboard and callback_query support (needed for Save/Discard buttons). node-telegram-bot-api would also work but is older and less actively maintained. Low effort to reverse — similar API surface. Bryan approved. |
| 2026-02-24 12:10 | Notion DB creation method (actual) | Via Notion API script (not MCP) | Originally planned MCP but used a direct Notion API script (`scripts/create-notion-dbs.js`) instead. Simpler — one `node scripts/create-notion-dbs.js` command creates both databases with all properties and select options. MCP would have been more complex to set up for a one-time operation. |
| 2026-02-24 | Defer voice notes (Whisper) to Phase 2 | Skip for MVP | Not a core need right now. Saves an OpenAI API key setup, one npm dependency, and a separate billing account. Notes module keeps Voice as a type — pipeline can be added later by adding `openai` package + `OPENAI_API_KEY` env var. Low effort to reverse. |
| 2026-02-24 | Runtime: Node.js or Python? | Node.js (single runtime) | Both specs list Node and Python deps side-by-side without choosing. Node wins: specs lean Node-first (@notionhq/client, better-sqlite3, node-cron all referenced first). Telegram bot ecosystem (grammY) is mature. Single runtime = one process, one deployment, simpler VPS management. |
| 2026-02-24 | Voice transcription provider (when ready) | OpenAI Whisper API | Self-hosting whisper.cpp is CPU-heavy and slow on a small VPS. OpenAI Whisper API costs ~$0.006/min — negligible for personal use. Groq offers Whisper at lower cost with faster response — viable swap later if needed. |
| 2026-02-24 | Buffer priority vs global intent engine | Draft buffer intercepts first when open | When a draft is open, all incoming messages go to the buffer's intent-shift detector first. Only if `continues_draft = false` does the message get released to the global intent engine. Natural UX: mid-note, you don't want "remind me" swallowed by the global router before the buffer can evaluate context. No draft open → global intent engine directly. |
| 2026-02-24 | Scheduler tables: unified or separate? | Unified `scheduled_jobs` table | Specs had 3 separate tables (recurring_tasks, reminders, briefing triggers) which means 3 background workers checking independently. Merged into one table with a `type` column and `cron_expr` for recurring. One worker, one query (`next_run_at <= now`), one loop. Simpler and easier to debug. |
| 2026-02-24 | Save/Discard: Claude intents or Telegram callbacks? | Telegram callback_query | CONFIRM_DRAFT and DISCARD_DRAFT were listed as Claude-classified intents in the Quick Notes spec. They're actually inline button taps, not natural language. Handling as callback_query is instant, free (no API call), and keeps the intent engine clean for NLP-only work. |
| 2026-02-24 | Stream routing: per-module or shared? | Shared streamRouter.js module | The keyword→stream mapping table existed only in the todo spec. Notes spec said "infer from content keywords" without defining which keywords. Extracted into one shared module that returns `{stream, confidence}`. Todos default to Personal on low confidence; Notes leave stream blank. Same router, different fallback behavior per caller. |
| 2026-02-24 | Daily briefing: monolithic or composable? | Composable (each module contributes a section) | Briefing needs tasks + reminders + calendar (Phase 2). Monolithic function would need rewriting when calendar module ships. Composable design: each module exports a `getBriefingSection()` function. Calendar placeholder returns empty array until spec is written. Extensible without touching existing code. |
| 2026-02-24 | Calendar module timing | Phase 2 (not in MVP) | Master plan includes Google Calendar + Outlook Calendar. No module spec exists. Daily briefing depends on calendar data but can function without it (placeholder section). Building without a spec risks rework. Calendar deferred; briefing ships with tasks + reminders only. |
| 2026-02-24 | LIST_WAITING: separate intent or LIST_TODOS filter? | Filter within LIST_TODOS (filter: "waiting") | LIST_WAITING was declared in the todo spec's intent list but had no JSON schema or behavior definition. It's functionally identical to `LIST_TODOS` with `filter: "waiting"`. Removed as a separate intent; handled by existing LIST_TODOS filter logic. |
| 2026-02-24 | Git branching: 4-branch or simplified? | 2-branch: dev + main (no staging) | ECOMWAVE has 7 users and a client — staging is essential. This is a single-user personal tool. Bryan tests on dev, approves, merges to main. Feature branches off dev as needed. No staging overhead. |
| 2026-02-24 | Notion DB creation: manual or via MCP? | Via MCP (Claude Code creates programmatically) | Claude Code can create both databases with all properties, select options, and views matching SCHEMA.md exactly. Faster and less error-prone than manual creation. Bryan just needs to share a Notion page with the integration. |
| 2026-02-24 | VPS port for Telegram Bots | Port 3003 | 3001 and 3002 taken by ECOMWAVE CRM (prod + staging). 3003 is next sequential, easy to remember. |
